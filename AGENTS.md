# ADAMANT Schema: AI Agent Operating Manual

This document defines how AI agents must work in this repository.

## Mission

`adamant-schema` is the **OpenAPI 3.0 specification source** for the ADAMANT Node API, served as a Swagger UI at <https://schema.adamant.im>.

Agent output must optimize for:

1. Correctness and completeness of the API specification
2. Backward compatibility of API contracts consumed by downstream tools
3. Simplicity — the codebase is lightweight; avoid over-engineering

## Language Policy

- Developers may communicate with AI in any language
- All repository artifacts (YAML, code, comments, docs, commit messages, PR text) must be in English only

## Project Structure

```
specification/
  openapi.yaml          # Root OpenAPI 3.0 spec — registers all schemas and paths
  accounts/             # Account management endpoints and DTOs
  blocks/               # Block info endpoints and DTOs
  chats/                # Chat messages, rooms, and transactions
  common/               # Shared params, properties, and transaction schemas
  delegates/            # Delegate and voting endpoints
  kvs/                  # Key-value storage endpoints
  nodes/                # Node status, network info, and blockchain metrics
  transactions/         # Transaction endpoints and DTOs
app.ts                  # Express + swagger-ui-express server (serves dist/schema.json)
dist/                   # Bundled output — generated by `npm run bundle`; do not hand-edit
examples/axios/         # Example: generating TypeScript client types from the schema
```

## Common Commands

```bash
# Install dependencies
npm install

# Bundle specification/ into dist/schema.json
npm run bundle

# Bundle and start Swagger UI at http://localhost:3000
npm run start

# Watch specification/ for changes and auto-rebuild
npm run start:watch

# Format with Prettier (also runs automatically on pre-commit via husky)
npx prettier --write .
```

## Ecosystem Role

This repository is the **source of truth for the ADAMANT Node API contract**. Changes here affect downstream consumers:

- [`adamant-api-jsclient`](https://github.com/Adamant-im/adamant-api-jsclient) — uses this schema to generate `src/api/generated.ts` TypeScript types
- [`adamant`](https://github.com/Adamant-im/adamant) — references this repo as the API schema companion artifact

When adding, removing, or renaming API fields or endpoints:

1. Confirm the behavior against the live node: <https://docs.adamant.im>
2. After merging, notify or update consumers — especially `adamant-api-jsclient`
3. If a field is being deprecated, preserve it in the schema until all consumers have migrated

## Validation and Formatting

There is no test framework. Validation is:

1. `npm run bundle` — must complete without errors
2. Visual check in Swagger UI — `npm run start`, then open <http://localhost:3000>

Formatting is handled by Prettier. It runs automatically on pre-commit (husky + lint-staged for `*.{json,yaml,js,ts}`). Prettier config: 2-space indent, single quotes, trailing commas (ES5), semicolons.

## Deployment

Pushing to `master` triggers the GitHub Actions workflow (`.github/workflows/github-pages.yaml`) which bundles and deploys to GitHub Pages → <https://schema.adamant.im>.

Work in `dev` and open PRs targeting `dev`. Merges to `master` are release actions.

## Issue, Label, and PR Conventions

Follow organization-wide conventions:

- Governance: <https://github.com/Adamant-im/.github>
- Title prefix guidance: <https://github.com/orgs/Adamant-im/discussions/5>
- Label catalog: <https://github.com/orgs/Adamant-im/discussions/1>

### Issue title prefixes (one or two maximum)

- `[Bug]` — incorrect or missing spec, wrong response shape
- `[Feat]` — new endpoint or schema component
- `[Enhancement]` — improvement to existing schema
- `[Docs]` — documentation updates
- `[Chore]` — dependency updates, config changes
- `[Task]` — general task

### Label policy

Apply a minimal but informative set: one type label (`Task`, `bug`, `enhancement`) + one or more domain labels (`APIs`, `NodeJS`, `TypeScript`, `documentation`, `Nodes`, etc.).

### PR title format

Use `Type: Short summary` — for example `Feat: Add timestampMs field to transaction schema`. No square brackets in PR titles.

### Branch and PR target

- Target `dev` for all PRs
- Branch naming: `type/short-description` — for example `feat/add-timestamp-ms`

### CLI workflow for multi-line content

Always use file-based flags and `.ai-ignored/` for temp files — avoid inline multi-line shell strings:

```bash
gh issue create \
  --body-file .ai-ignored/temp.YYYY-MM-DD.issue-body.md \
  --label "Task,documentation"

gh pr create \
  --body-file .ai-ignored/temp.YYYY-MM-DD.pr-description.md
```

`.ai-ignored/` is git-ignored. Use dated filenames; cleanup is optional.

## AI Change Workflow

1. Read `specification/openapi.yaml` and the relevant sub-directory YAML files before editing
2. Identify what downstream consumers rely on (field names, response shapes, required/optional flags)
3. Make the minimal necessary change
4. Run `npm run bundle` — fix any errors before proceeding
5. Visually verify the change in Swagger UI
6. Report what was changed, what was preserved, and any consumer impact

## Definition of Done

A change is done when:

- `npm run bundle` completes without errors
- Swagger UI renders the endpoint or schema correctly
- API contract is consistent with ADAMANT Node behavior (<https://docs.adamant.im>)
- Downstream consumer impact is noted in the PR description
- All repository artifacts are in English
